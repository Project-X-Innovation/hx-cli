# Tech Research: Library Comments and Iteration — helix-cli

## Technology Foundation

- **Runtime**: Node.js (pure TypeScript, no external CLI framework)
- **Build**: `tsc` (TypeScript compilation)
- **HTTP Client**: Custom `hxFetch` wrapper with API key/bearer auth, retry logic (`src/lib/http.ts`)
- **Routing**: Switch-based command dispatch in `src/index.ts`
- **Config**: Multi-org config in `~/.hlx/config.json` with HELIX_API_KEY/HELIX_URL env fallbacks

## Architecture Decision

### Decision: New src/library/ Module (not extension of comments)

**Options considered:**

| Option | Description | Pros | Cons |
|--------|-------------|------|------|
| **A. Extend comments module** | Add `--library` flag to existing `hlx comments` commands | Reuses existing code paths; fewer new files | Awkward UX (`hlx comments post --library --item X`); conflates two domains; comments module is tightly coupled to --ticket flag |
| **B. New library module (chosen)** | New `src/library/` directory with index.ts router and subcommand files | Clean domain separation; intuitive command structure (`hlx library comments post`); follows existing module pattern | New files to create; some pattern duplication with comments module |

**Chosen option: B (New library module)**

**Rationale**: Library items and ticket comments are distinct domains with different API endpoints, reference resolution, and interaction patterns. The existing comments module (src/comments/) is tightly coupled to `--ticket` flag resolution and `/api/tickets/{ticketId}/comments` endpoints. A new library module provides intuitive command hierarchy (`hlx library list`, `hlx library show`, `hlx library comments list/post`) consistent with how tickets and comments are separate modules today. The module follows the exact same structural pattern: index.ts router, subcommand files, flag parsing via lib/flags.ts.

### Decision: Heading Slug Section Targeting (not line numbers or text)

**Options considered:**

| Option | Description | Pros | Cons |
|--------|-------------|------|------|
| **A. Line numbers** | `--section 15-30` targets content by line range | Precise; familiar for code | Not meaningful for markdown; changes with formatting; users don't know line numbers |
| **B. Heading slugs (chosen)** | `--section "key-findings"` targets by slug | Stable; matches server anchor format; human-readable; discoverable via `hlx library show` | Requires slug knowledge; limited to heading-level |
| **C. Text match** | `--section "Key Findings"` matches heading text | Most human-friendly; flexible | Ambiguous with partial matches; case sensitivity; needs fuzzy matching |

**Chosen option: B (Heading slugs)**

**Rationale**: The `anchor` field in the server's LibraryComment model stores heading slugs generated by rehype-slug. Using the same slugs in the CLI ensures a direct mapping without translation. The `hlx library show` command displays section headings with their slugs, making them discoverable. Slugs are stable across minor text changes (rehype-slug is deterministic). For human convenience, the implementation can also accept the exact heading text and slugify it client-side as a fallback, but the slug is the primary interface.

## Core API/Methods

### Command Structure

```
hlx library list                              List library items
hlx library show <ref>                        Show report content with section headings/slugs
hlx library comments list <ref>               List all comments on a library item
hlx library comments list <ref> --section <slug>   List comments for a specific section
hlx library comments post <ref> --section <slug> --rating <value> [message]
```

### Module Files

| File | Purpose |
|------|---------|
| `src/library/index.ts` | Router: dispatches to list, show, comments subcommands |
| `src/library/list.ts` | `hlx library list` — GET /api/library/items, tabular output |
| `src/library/show.ts` | `hlx library show <ref>` — GET /api/library/items/:id, display content with section headings |
| `src/library/comments.ts` | Nested router: dispatches to comments-list, comments-post |
| `src/library/comments-list.ts` | `hlx library comments list <ref>` — GET /api/library/items/:id/comments |
| `src/library/comments-post.ts` | `hlx library comments post <ref>` — POST /api/library/items/:id/comments |
| `src/lib/resolve-library-item.ts` | Resolve item by ID, ticket short ID, or title match |

### Item Resolution: resolve-library-item.ts

Following `resolve-ticket.ts` pattern:
1. If input is a cuid (starts with "c", 25 chars), treat as library item ID directly
2. If input matches short ID pattern (e.g., "RSH-439"), use `GET /api/library/items?ticketId=` to find the latest item for that ticket
3. Fallback: list all items, match by title substring (case-insensitive)

This requires the server's existing `/api/library/items?ticketId=` endpoint to resolve short IDs to ticket IDs first (via `GET /api/tickets` or a dedicated lookup). The resolve function can use the existing ticket resolution to get the ticketId, then look up the library item.

### Rating Flag

`--rating` accepts human-readable values that map to server enum:
- `thumbs-up` -> `THUMBS_UP`
- `love` -> `LOVE`
- `thumbs-down` -> `THUMBS_DOWN`

CLI also accepts shorthand: `up`, `down`, `love`.

### Output Formatting

**`hlx library list`**: Tabular output similar to `hlx tickets list`:
```
ID          Title                    Status      Date
cm...abc    Market Analysis Report   PUBLISHED   2026-05-10
cm...def    Competitor Deep Dive     DRAFT       2026-05-12
```

**`hlx library show <ref>`**: Full report content with section headings annotated with slugs and comment summaries:
```
# Market Analysis Report (cm...abc)

## Introduction [introduction] (2 comments: 1 thumbs-up, 1 love)
[section content...]

## Key Findings [key-findings] (3 comments: 1 thumbs-up, 1 thumbs-down, 1 love)
[section content...]
```

**`hlx library comments list <ref>`**: Comments grouped by section:
```
## introduction (2 comments)
  [thumbs-up] Alice (2026-05-10): "Great framing"
  [love] Bob (2026-05-11)

## key-findings (3 comments)
  [thumbs-down] Alice (2026-05-10): "Wrong level of abstraction"
    -> [reply] Bob (2026-05-11): "I disagree, this is the right level"
  [thumbs-up] Carol (2026-05-11): "Dive deeper into this"
```

## Technical Decisions

### Separate Module (not Flag Extension)
- **Chosen**: New `src/library/` module
- **Rejected**: Adding `--library` flag to `hlx comments` — conflates domains, breaks existing command interface
- **Evidence**: src/comments/index.ts is hard-coupled to ticket resolution

### Slug-Based Section Targeting (with Text Fallback)
- **Chosen**: `--section` flag with heading slug as primary value
- **Enhancement**: If the value contains spaces or capitals, slugify it before sending to API (e.g., `--section "Key Findings"` -> anchor: "key-findings")
- **Rejected**: Positional argument for section — would conflict with message text
- **Evidence**: rehype-slug uses GitHub-flavored slugification (lowercase, hyphens)

### Human-Readable Rating Values (not Enum Values)
- **Chosen**: `--rating thumbs-up` (with shorthand `up`/`down`/`love`)
- **Rejected**: `--rating THUMBS_UP` — unfriendly for human CLI use
- **Mapping**: CLI translates to server enum values before API call

## Cross-Platform Considerations

- **Agent usage**: Agents interact via SKILL.md documentation. Command structure should be predictable and machine-parseable. All outputs include IDs for programmatic chaining.
- **Pipe-friendly**: `hlx library list` outputs to stdout in a format that can be piped/parsed. Consider `--json` flag for machine-readable output.
- **Error handling**: Consistent with existing CLI patterns — `console.error("Error: ...")` + `process.exit(1)`.
